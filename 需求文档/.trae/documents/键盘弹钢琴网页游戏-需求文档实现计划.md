# 键盘弹钢琴网页游戏 — 需求文档实现计划

> **用途**：为 TRAE AI 创意大赛（生活娱乐赛道）的「键盘弹钢琴网页游戏」创意，产出一份 HTML 格式的需求文档（PRD）。文档双重用途：① 可提炼为报名创意提案（报名截止 7 月 15 日）；② 作为初赛 Demo 的开发规格。文档除完整需求外，专设"缺失与完善建议"章节，直接回应用户"指出疏漏"的诉求。
> **执行者须知**：本计划为决策完整的蓝图，执行时不再做新的产品决策，只按蓝图撰写 HTML。最终交付用 `html-report` 技能生成自包含 HTML。

## 一、概述（Summary）

产出一份 HTML 需求文档，描述一款「键盘弹钢琴网页游戏」：用电脑键盘 / 手机触屏弹钢琴，支持**跟弹打分**与**自由弹奏**双模式；可上传本地 MP4 由**浏览器端 AI**（Spotify Basic Pitch）自动转乐谱与按键提示；配 Three.js 3D 钢琴 / 粒子 / 二次元背景与摄像头画中画的沉浸视觉，风格可切换；含历史记录与歌单推荐；桌面 + 手机全适配。受众男女老少，目的放松娱乐。

文档遵循重型 PRD 规范（0-to-1 产品），功能需求详述占 50%+ 且每个模块嵌入可交互原型 Demo，并附竞品分析、AI 专项、技术可行性与 19 项缺失/完善建议。

## 二、当前状态分析

### 大赛背景
- 赛事：TRAE AI 创造力大赛；赛道四选一，用户选「生活娱乐」。
- 报名：2026-06-16 ~ 07-15，需发创意提案帖并附 TRAE Work 生成的创意产物 HTML。
- 初赛：把创意做成可运行、可体验的 Demo 提交；提交形式可为「交互式可体验的 HTML 压缩包」，**无需部署上线**；评审标准为能正常体验核心功能、理解核心价值即达标。

### 已确认的产品决策（来自用户）
- 核心玩法：跟弹打分 + 自由弹奏（双模式）。
- 平台：桌面 + 手机全适配。
- 追加功能：历史记录、歌单推荐。

### 技术可行性结论（已联网调研，非假设）
- **发声**：Web Audio API 浏览器原生，低延迟，支持多音。
- **AI 转谱**：Spotify Basic Pitch（`@spotify/basic-pitch`，TS 库，Apache-2.0）可纯浏览器端运行，输入 AudioBuffer（mp3/ogg/wav/flac），输出 notes/MIDI，支持多音与音高弯曲。无需后端，契合"HTML 压缩包"提交形式与隐私要求。
- **MP4**：Basic Pitch 不直接处理 MP4 视频容器，需先提取音轨（ffmpeg.wasm 懒加载，或引导用户直接上传音频文件）。
- **乐谱渲染**：OpenSheetMusicDisplay（基于 VexFlow）在浏览器渲染 MusicXML 五线谱。
- **3D 视觉**：Three.js 成熟可用。
- **竞品**：Synthesia（下落音符 + 旋律练习 + 读谱切换 + MIDI/MusicXML 导入）、Virtual Piano（字母键映射，年活 1900 万）、OnlinePianist（动画教程）。三者均无"上传音频 → AI 转谱"能力，也无 3D 沉浸视觉——构成本产品的差异化点。

### 竞品分析要点
竞品矩阵（Synthesia / Piano Tiles·Magic Tiles / Virtual Piano / OnlinePianist / 本产品）沿"形态、核心机制、输入、AI 能力、乐谱渲染、沉浸视觉、定位"七维对比。差异化机会三条：① 唯一具备"上传本地音频 → 浏览器端 AI 转谱 → 自动生成乐谱与按键提示"；② 唯一结合 Three.js 3D 钢琴 + 粒子 + 二次元背景 + 摄像头画中画的沉浸视觉；③ 纯前端零后端、单 HTML 压缩包提交，隐私友好（音频不出本机）。

### 功能模块清单（M01–M14）
每模块统一格式：编号 / 名称 / 触发→处理→结果链 / 交互原型 Demo / 验收标准 / 边界与降级。

| 编号 | 名称 | 触发→处理→结果链（要点） | 交互原型 Demo | 关键验收 |
|---|---|---|---|---|
| M01 | 音源引擎 | 键盘/触屏/自动播放触发 → 查 MIDI→频率表 → 选音色 → Web Audio 触发 + ADSR 包络 → 即时发声 + 按键高亮 | 可点击小钢琴（7 白 + 5 黑）+ 音色下拉，点键出声显音名 | 按下到出声 < 50ms；同时 4 键无爆音丢音 |
| M02 | 自由弹奏 | 选"自由弹奏" → 加载音源 + 输入监听 → 实时映射发声 + 3D 键动画 + 粒子 | 钢琴 + 按键提示浮层 + 八度切换 + 录音按钮 | 任意键即时发声；八度切换映射正确；触屏多点响应 |
| M03 | 跟弹打分 | 选曲选难度 → 按时间轴生成下落音符 → 到判定线按键判定（Perfect/Great/Good/Miss）→ 累分 + 连击 + 特效 → 结算页写入历史 | 竖向下落区，A/S/D 命中显判定文字 + 分数跳动 + "自动演示"按钮 | 判定轴与谱面对齐误差 < 30ms；连击逻辑正确 |
| M04 | AI 转谱 | 上传 MP4/音频 → 提取音轨(M04a) → Basic Pitch 推理(M04b) → notes→MIDI+MusicXML → OSMD 渲染 + 下落谱面 | 上传按钮 + 进度条 + 结果区（五线谱缩略 + 下落音符预览） | 30s 单乐器音频 < 60s 完成；结果可进跟弹；失败有降级提示 |
| M05 | MIDI→按键映射 | 加载谱面 → 读 MIDI 编号 → 映射到可见八度琴键 → 超出则平移/折叠 → 生成键位字母提示 | "琴键↔字母↔MIDI 编号"三对照表，点高亮联动小钢琴 | C 大调音阶默认八度全可弹；3 音和弦映射到 3 个物理键 |
| M06 | 乐谱展示 | 开启"显示乐谱"/转谱完成 → MusicXML→OSMD 渲染 → 跟弹时高亮当前小节 | 固定 MusicXML 渲染区块 + 放大/缩小/隐藏按钮 | 标准 MusicXML 正常渲染；高亮跟随播放无卡顿 |
| M07 | 历史记录 | 跟弹结算/录音结束 → 写 localStorage/IndexedDB → 历史页列表 + 进步曲线 | 历史卡片列表 + 近 10 次分数折线图 | 刷新后数据保留；存储超限有提示 |

| M13 | 全平台适配 | 响应式 + 移动触屏琴键（可滑八度）+ 横屏优先 | 桌面/移动两套布局缩略 + 触屏琴键 ≥44px | 横竖屏自适应；触屏多点可同时响应 |
| M14 | 录音回放 | 自由弹奏点录音 → 记录 {midi,time,duration,velocity} → 回放走同一音源 → 可导出 MIDI | 录音按钮 + 回放 + 导出 .mid | 回放音序正确；导出文件可被其他工具打开 |

### AI 专项要点
- **能力边界**：Basic Pitch 单乐器清晰录音效果最佳；复杂混音（人声+伴奏+多乐器）准确率显著下降。定位"娱乐向自动转谱"，UI 明示"AI 识别结果仅供参考"。
- **运行位置**：100% 浏览器端，模型懒加载，音频不出本机。
- **分级降级**：模型加载失败 → 引导用内置曲库；推理超时 → 可继续等待/取消；复杂混音 → 提示音符偏少建议更清晰音频；不支持 WebAssembly → 进入无 AI 基础体验。
- **输入约束**：支持 MP4（先提取音轨）/WAV/MP3/OGG/FLAC；建议 ≤ 2 分钟、≤ 10MB；自动降混单声道、重采样 22050Hz。

### 非功能需求（表格）
性能（首屏<3s、判定延迟<30ms、出声<50ms、桌面60fps/移动30fps）、兼容（Chrome/Edge 最新 2 版、Safari 16+、移动 Chrome/Safari；iOS 首次交互激活 AudioContext）、隐私（摄像头/音频均纯本地不上传、无网络请求）、体积（< 30MB 含模型、懒加载）、可靠性（断网可用、存储超限兜底）、可维护性（模块化/TS/配置驱动）。

### 技术可行性与架构
- **选型总表**：发声 Web Audio API（+ Tone.js/SoundFont 采样）；AI `@spotify/basic-pitch`；MP4 提取 ffmpeg.wasm 懒加载；乐谱 OpenSheetMusicDisplay；3D Three.js；存储 IndexedDB+localStorage；构建 Vite+TS。
- **分层架构**：表现层(UI) → 交互层(输入路由/判定器/反馈调度) → 领域层(音源/AI 流水线/谱面解析/打分/历史) → 基础层(Web Audio/Three.js/OSMD/ffmpeg.wasm/Basic Pitch/Storage)。
- **三条关键数据流**：弹奏链（输入→映射→发声+3D动画+粒子）、跟弹链（谱面→时间轴→下落音符→判定→打分→历史）、AI 链（MP4→音轨→AudioBuffer→Basic Pitch→notes→{MIDI,MusicXML}→{下落谱面,OSMD 乐谱}）。
- **体积控制**：模型单独懒加载；ffmpeg.wasm 仅 MP4 路径按需加载；Three.js 按需引入；曲库 MIDI 文本/Base64 内嵌。

## 五、关键决策与假设

### 决策
| # | 决策 | 选择 | 理由 |
|---|---|---|---|
| D1 | AI 运行位置 | 纯前端 Basic Pitch TS | 满足 HTML 压缩包 + 隐私 + 零后端 |
| D2 | 音源方案 | 小体积 SoundFont 采样(钢琴) + 合成音色(音乐盒/Pad) | 兼顾音质与体积 |
| D3 | MP4 提取 | 引导直传音频优先 + ffmpeg.wasm 单线程懒加载兜底 | 规避 SharedArrayBuffer 头要求 |
| D4 | 乐谱渲染 | OpenSheetMusicDisplay | 官方支持 MusicXML |
| D5 | 打分窗口 | Perfect±50/Great±100/Good±150ms | 平衡难度与娱乐 |
| D6 | 键盘映射 | 双行映射 + Z/X 切八度 | 88 键折叠到可用按键 |
| D7 | 数据存储 | IndexedDB + localStorage | 纯前端持久化 |
| D8 | 摄像头 | 默认关 + 首次隐私弹窗 | 降低敏感度 |
| D9 | 移动端 | 横屏优先 + 触屏琴键 | 触屏体验 |
| D10 | 提交形态 | 单 HTML 压缩包，资源内联/同包 | 大赛要求 |

### 假设
- A1：评委环境为现代桌面浏览器（Chrome/Edge），支持 WebAssembly 与 WebGL。
- A2：Basic Pitch 模型可在本地静态服务器/file:// 下加载（执行时验证跨域与模型路径）。
- A3：公有领域/自创曲库可随包分发，无版权风险。
- A4：用户上传 MP4 为本地文件，提取与推理均在本地完成。
